## Main Conversion Goal

**Get users from raw prompt idea → saved, versioned, evaluated prompt in the library as fast as possible.**

The critical conversion funnel is:

```
Open Prompt Lab → Type/paste prompt → Enhance → Review output → Save to library
```

Every interaction should either accelerate this funnel or provide a high-value side loop (compare, test, iterate) that feeds back into it.

---

## Three Most Important Workflows

### 1. Enhance & Save (the core loop)
```
Type raw prompt → Select mode → Cmd+Enter → Review enhanced output → Cmd+S → Title/tags → Save
```
**Frequency:** Every session, multiple times per session
**Current steps:** 6-8 interactions
**Pain:** Save panel is separate from the flow — user has to context-switch to fill metadata

### 2. Iterate & Compare (the quality loop)
```
Edit prompt → Re-enhance → Compare to previous version or golden → Adjust → Save new version
```
**Frequency:** Every serious prompt engineering session
**Pain:** Version comparison requires opening a modal; golden diff is inline but not visually prominent; no quick "try again with different mode" action

### 3. Library Browse & Reuse (the value-capture loop)
```
Search/filter library → Find prompt → Copy or Load into editor → Modify → Use
```
**Frequency:** Growing as library size grows
**Pain:** Library is a sidebar — compact card view hides most content; expanding a card pushes everything else down; no quick-copy without expanding

## Known Pain Points (from QA, code analysis, and structural review)

1. **Modal stacking risk:** Template Variables, Settings, Shortcuts, Version Diff, and PII Warning all use `fixed inset-0` — no modal queue prevents overlap
2. **Stale editingId cross-contamination:** Promoting from Scratchpad while editing a library item can cause the next save to overwrite the wrong entry (QA test 8.2)
3. **Cmd+T / Cmd+W conflict:** Scratchpad shortcuts hijack browser new-tab and close-tab — hostile in extension context
4. **Information density overload:** Expanded library cards show test cases, versions, original/enhanced, notes, tags, and actions in a single scrolling column — no progressive disclosure hierarchy
5. **No visual indicator of unsaved work:** User can navigate away from editor with unsaved enhanced output and lose it
6. **Comma-separated trait entry:** Test case expected traits and exclusions are raw comma-separated strings, not visual chips — error-prone and hard to scan
7. **Copy feedback is toast-only:** No inline button state change ("Copied!") — toast can be missed in peripheral vision
8. **Run history has no filter persistence:** Filters reset on tab switch — returning users must re-apply
9. **Command palette is shallow:** Only ~5 global commands; doesn't index library entries, modes, or recent prompts
10. **No "try again" shortcut:** After enhancing, switching mode and re-running requires 2 interactions instead of a one-click "Re-enhance with [mode]" action

## 1. Executive Summary

Prompt Lab is a **technically mature, feature-rich prompt engineering workbench** with strong fundamentals: local-first persistence, five-provider support, deterministic versioning, golden benchmarks, and professional-grade keyboard shortcuts. The design system is clean (violet accent, dark-first, density-adaptive) and the accessibility foundation is solid (ARIA labels, focus rings, reduced-motion support).

However, the product has grown feature-by-feature without a corresponding IA restructure. The result is a **5-tab flat navigation** that conceals the relationship between workflows, a **library sidebar that tries to do too much in too little space**, and **several high-value features (golden benchmarks, version history, test cases) that are buried 2-3 clicks deep inside expandable cards**.

The primary risk is not missing features — it's that the features that exist are **hard to discover, hard to reach, and easy to lose context in**. This is a product where the power is real but the surface doesn't communicate it.

**Strategic recommendation:** Consolidate the 5-tab model into a 3-surface model (Create, Library, History), promote test cases and golden benchmarks to first-class panels, and invest in a richer command palette as the expert-user accelerator.

## 3. Current Information Architecture Problems

### Problem 1: Flat 5-tab model doesn't reflect workflow hierarchy

The current tabs are:
```
[ Editor ] [ Composer ] [ A/B ] [ Pad ] [ History ]
```

This treats 5 features as equal peers, but they have fundamentally different roles:

| Tab | Role | Usage Frequency |
|-----|------|-----------------|
| Editor | Primary creation surface | Every session |
| Composer | Composition tool | Occasional |
| A/B Tester | Evaluation tool | Occasional |
| Scratchpad | Scratch/notes | Frequent but secondary |
| Run History | Analytics/review | Periodic |

**Consequence:** Users must memorize which tab holds which capability. New users don't know where to start. The "primary loop" (enhance → save → reuse) is spread across Editor + Library sidebar, creating spatial fragmentation.

### Problem 2: Library is demoted to a sidebar

The library is the **persistent value store** of the entire product — every prompt, version, test case, and golden benchmark lives here. But it's rendered as a collapsible sidebar panel that shares horizontal space with the editor.

In compact/extension mode, the library is hidden entirely and requires a toggle to reveal.

**Consequence:** The library feels like an afterthought rather than a core surface. Discovery, search, and management are cramped.

### Problem 3: Deep nesting hides high-value features

The feature hierarchy looks like:
```
Tab → Library sidebar → Card → Expand card → Test Cases section → Individual test case
Tab → Library sidebar → Card → Expand card → Versions section → Version diff modal
Tab → Enhanced output → Golden response section → Threshold slider
```

Three of the product's most differentiating features (golden benchmarks, version history, test cases) are 3-4 clicks deep from any entry point.

### Problem 4: No persistent workspace context

There's no concept of "I'm working on this prompt" that persists across tab switches. If a user enhances a prompt in Editor, switches to History to check a previous run, and comes back — they have to re-orient. The editor doesn't show "Working on: [prompt title]" or breadcrumbs.

## 4. Proposed Revised Information Architecture

### Model: 3 primary surfaces + command palette as accelerator

```
Prompt Lab
├── [Surface 1] WORKBENCH ←── merge Editor + Composer + Scratchpad
│   ├── Editor mode (default)
│   │   ├── Input pane
│   │   ├── Enhanced output pane
│   │   ├── Inline save bar (collapsed by default, expands on first enhance)
│   │   └── Quick actions rail (mode switcher, re-enhance, copy, save)
│   ├── Composer mode (toggle within Workbench)
│   │   ├── Block canvas
│   │   └── Live preview
│   └── Scratchpad mode (toggle within Workbench)
│       ├── Multi-pad editor
│       └── Promote → Editor transition
│
├── [Surface 2] LIBRARY ←── promoted to full surface
│   ├── Search + filter bar (sticky)
│   ├── Prompt cards (grid or list view toggle)
│   ├── Card detail panel (slide-out or inline expand)
│   │   ├── Versions timeline (inline, not modal)
│   │   ├── Test cases panel
│   │   ├── Golden benchmark status
│   │   └── Quick actions (Edit, Copy, A/B Test, Delete)
│   └── Collections sidebar (collapsible)
│
├── [Surface 3] EVALUATE ←── merge A/B + Run History
│   ├── Run Timeline (default view)
│   │   ├── Persistent filters
│   │   ├── Golden trend chart
│   │   └── Run cards with inline compare
│   ├── A/B Test mode (toggle within Evaluate)
│   │   ├── Variant editors
│   │   ├── Side-by-side results
│   │   └── Winner selection
│   └── Compare mode (2-run detail diff)
│
├── [Global] Command Palette (Cmd+K) — deep-indexed
├── [Global] Settings
└── [Global] Keyboard shortcuts
```

### Navigation change:

```
Before:  [ Editor ] [ Composer ] [ A/B ] [ Pad ] [ History ]   (5 equal tabs)
After:   [ Workbench ▾ ] [ Library ] [ Evaluate ▾ ]            (3 surfaces, 2 with sub-modes)
```

**Benefits:**
- Reduces top-level decision points from 5 → 3
- Groups by intent (Create / Store / Assess) not by feature
- Promotes Library to equal status with creation
- Keeps all sub-modes accessible via dropdown or keyboard, no feature loss
- Extension compact mode only needs to manage 3 surfaces instead of 5

## 5. Recommended Interaction-Pattern Changes

### 5.1 — Inline Save Bar (replaces separate save panel)

**Current:** User clicks Save → save panel slides open below enhanced output → user fills title/tags/collection → clicks Save button.

**Proposed:** After first enhancement, a compact save bar appears between the enhanced output and lint panel:

```
┌──────────────────────────────────────────────────────┐
│ 💾 [Auto-title from prompt...  ] [tags ▾] [Save ⌘S] │
└──────────────────────────────────────────────────────┘
```

- Title auto-populated from prompt text (already implemented as `suggestTitleFromText`)
- Tags as a compact chip selector (click to expand full tag panel)
- One-click save for the common case; expand for full metadata
- Stays visible until explicitly dismissed or content changes

### 5.2 — "Re-enhance" Quick Action

After enhancement completes, show mode pills directly in the output area:

```
Enhanced via Balanced ·  [Try Claude] [Try Concise] [Try Detailed]
```

Single click re-runs with different mode. No need to scroll up, change dropdown, and re-click Enhance.

### 5.3 — Deep Command Palette

Expand the command palette to index:
- All library entries (by title, tag, collection)
- All enhancement modes
- Recent prompts (last 5)
- Provider switching
- Density/theme toggles
- Tab/surface switching
- "Open test cases for [prompt]"
- "Compare last 2 runs"
- "Pin golden response"

This makes Cmd+K the expert-user expressway — anything reachable by mouse should also be reachable by typing 3-5 characters.

### 5.4 — Working Context Indicator

When a library prompt is loaded into the editor (via Edit or Load), show a persistent breadcrumb:

```
Working on: "Customer Support Rewriter" (v3) · [Detach] [View in Library]
```

This prevents the "which prompt am I editing?" disorientation and makes it clear whether Save will create or update.

### 5.5 — Verdict Cycling → Explicit Buttons

**Current:** Clicking the verdict badge cycles through null → pass → fail → mixed. This is a hidden gesture.

**Proposed:** Show explicit small buttons: `[✓ Pass] [✗ Fail] [~ Mixed]`. Active state highlighted. Same space, zero hidden interaction.

### 5.6 — Chip Input for Test Case Traits

Replace comma-separated text inputs for expected traits and exclusions with chip/tag inputs:
- Type a trait → press Enter or comma → chip appears
- Click X on chip to remove
- Visual scanning is immediate
- Same pattern already exists for prompt tags (TagChip component)

## 6. Visual Design Recommendations

### 6.1 — Visual Hierarchy Tightening

**Issue:** Most UI text is `text-xs` or `text-sm` with `uppercase tracking-widest` section labels. This creates a flat hierarchy where everything looks equally important.

**Recommendation:**
- Reserve `uppercase tracking-widest` for top-level section headers only
- Use `font-medium text-sm` for sub-section labels (not uppercase)
- Increase enhanced output font size by 1 step (currently same size as input — should feel like "the result")
- Add subtle left-border accent to the enhanced output pane (similar to blockquote treatment) to visually separate "what you wrote" from "what the AI returned"

### 6.2 — Status Density in Library Cards

**Issue:** Collapsed library cards show: title, preview text, date, use count, version count, variable count. This is 6 pieces of metadata on a card that's ~200px tall in compact mode.

**Recommendation:**
- Show only: title + first line of enhanced + primary tag
- Move metadata (use count, version count, date) to a hover tooltip or the expanded view
- Add a 2px left-border color-coded by collection (visual grouping without text)

### 6.3 — Enhanced Output Visual Distinction

**Issue:** The enhanced output textarea uses the same visual treatment as the input textarea. Users glance between them and lose track of which is which.

**Recommendation:**
- Add a subtle background tint to the enhanced pane (e.g., `bg-violet-950/20` in dark mode)
- Or add a thin top-border accent line in violet
- The output should feel like "the product" — slightly elevated or highlighted compared to the input

### 6.4 — Toast → Inline Feedback

**Issue:** Toasts at bottom-center are the only feedback for copy, save, delete, and promote actions.

**Recommendation:**
- **Copy:** Button text changes to "Copied!" for 1.5s (inline feedback)
- **Save:** Button briefly shows checkmark + "Saved" (inline feedback)
- **Delete:** Remain as toast (destructive confirmation should be prominent)
- **Promote:** Keep toast (cross-tab action needs global feedback)

### 6.5 — Empty States

**Issue:** Run History, Library (before any saves), and Composer Canvas have minimal empty states.

**Recommendation:** Each empty state should have:
- An icon (not decorative — functional, showing what belongs here)
- One sentence explaining what this area is for
- A single action button ("Enhance your first prompt" / "Save a prompt to start your library" / "Drag prompts here to compose")

## 7. Accessibility & Responsiveness Recommendations

### Accessibility

| Area | Current | Recommendation | Priority |
|------|---------|----------------|----------|
| **Focus management on tab switch** | No focus redirect | Move focus to first interactive element in new tab panel | High |
| **Modal focus trapping** | Not implemented | Add focus trap to all modals (Settings, Version Diff, PII, Variables) | High |
| **Screen reader tab announcements** | Tabs have `role="tab"` | Add `aria-controls` pointing to panel ID; add `role="tabpanel"` on content | High |
| **Verdict cycling** | No announcement | Add `aria-live="polite"` region for verdict changes; replace hidden cycle with explicit buttons | Medium |
| **Color-only status indicators** | Golden score bar is color-coded | Add text label alongside color (already partially done with % display) | Medium |
| **Link/button distinction** | Some `<span onClick>` patterns | Ensure all clickable elements are `<button>` or `<a>` | Medium |
| **Skip navigation** | Not present | Add "Skip to main content" link for keyboard users | Low |
| **Form error association** | Errors shown as text | Add `aria-describedby` linking error messages to inputs | Low |

### Responsiveness

| Area | Current | Recommendation | Priority |
|------|---------|----------------|----------|
| **Compact threshold** | 720px width OR 560px height | Consider content-based breakpoints; 720px may be too aggressive for tablets | Medium |
| **Library in compact mode** | Hidden, requires toggle | Show a floating action button or bottom sheet trigger for quick library access | High |
| **Composer on mobile** | Tab switcher (Library/Canvas/Preview) | Good pattern; keep. Consider swipe gestures between panes | Low |
| **A/B on mobile** | Tab switcher (A/B) | Add visual indicator showing which variant is currently visible | Medium |
| **Touch targets** | Minimum ~1.6rem in compact mode | Ensure all interactive elements meet 44×44px minimum on touch devices | High |

## Assumptions Made

1. **Extension side panel is the primary surface** — I optimized for narrow-viewport interactions first, desktop second
2. **Expert users are the primary audience** — recommendations favor keyboard-driven efficiency over onboarding wizards
3. **Library size will grow** — recommendations assume users will accumulate 50-200+ prompts and need real search/filter/sort
4. **Current React architecture is preserved** — no framework changes proposed, only component restructuring
5. **v1.x constraint holds** — no backend dependencies introduced; all changes are client-side
6. **Screenshots (4 captures from Mar 14) exist** but were inaccessible during this audit — visual polish recommendations are based on code-level analysis of CSS tokens, Tailwind classes, and component structure
