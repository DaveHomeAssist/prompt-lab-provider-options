auth-execution-packet.md
Date: 2026-03-23

Objective
Implement the next monetization and hosted-auth phases for Prompt Lab without destabilizing the shared-core architecture used by the hosted web app, extension, and desktop shells.

Guiding constraints
- Preserve the shared React core in `prompt-lab-extension/src`.
- Keep auth scoped to `promptlab.tools` hosted web only.
- Do not extend auth into extension or desktop in this phase.
- Treat library recovery as closed-out infrastructure that must remain working.
- Keep all Stripe work in test mode until the full loop is verified.

Phase 0: Baseline and Closeout Verification
Timeline: 0.5 day
Owner: Lead engineer
Support: QA/dev verifier

Goals
- Confirm the current recovery path and Node/runtime assumptions remain healthy before new work lands.
- Freeze the current implementation baseline so auth and billing work are not layered on top of unknown regressions.

Key milestones
- Verify hosted library recovery still works on the live web app.
- Verify Node 22 is active locally and in deploy config.
- Record a clean baseline of current auth-related files and missing API routes.

Required resources
- Local repo checkout
- Vercel project access
- Node 22 runtime
- Browser for manual recovery checks

Deliverables
- Baseline verification note in repo docs or session log
- Confirmed current-state inventory of existing auth shell files and absent billing endpoints

Phase 1: Hosted Web Auth Implementation
Timeline: 1 day
Owner: Lead engineer
Support: UI engineer/design reviewer

Goals
- Add hosted-web-only authentication with Clerk.
- Keep shared-core app behavior intact outside the web shell.

Key milestones
- Wrap the hosted web entry point in `ClerkProvider`.
- Add sign-in and sign-up flow with Clerk prebuilt components.
- Add normalized `useCurrentUser()` wrapper.
- Confirm safe fallback behavior when Clerk config is absent in local/dev contexts.

Required resources
- Clerk application and publishable key
- `.env.local` / Vercel env configuration
- Existing hosted app shell under `prompt-lab-web/src`

Responsible parties
- Lead engineer: implementation
- Product owner: confirm web-only auth scope
- Design reviewer: validate auth screen fit with Prompt Lab visual system

Deliverables
- Hosted auth shell wired into the web app
- Current-user abstraction with owned shape
- Updated docs for auth env vars and behavior

Acceptance criteria
- User can sign in and sign up on the hosted web app.
- Signed-in session survives refresh.
- Shared app still renders correctly in extension and desktop without Clerk.

Critical dependencies
- Clerk must be working before Stripe checkout work begins.
- Stripe identity binding depends on the hosted auth model.
- Entitlements depend on Stripe webhook events and a chosen storage model.
- Final verification depends on all prior phases being deployed to a testable environment.

Top risks
- Hosted auth shell diverges from shared-core assumptions and introduces regressions.
- Stripe endpoints are added before identity/state handling is stable.
- Entitlement state is treated as authoritative before storage and webhook reliability are proven.
- Documentation overstates what is implemented, causing future confusion.

Recommended milestone checkpoints
- Checkpoint 1: Auth shell working in preview deploy
- Checkpoint 2: Stripe checkout and portal endpoints reachable in test mode
- Checkpoint 3: Entitlement updates observable from webhook to client hook
- Checkpoint 4: Full subscribe/cancel verification complete
- Checkpoint 5: Docs and handoff updated

Definition of done
- Hosted web auth is live and scoped correctly.
- Stripe scaffold works in test mode.
- Entitlement state can be queried and used safely.
- Recovery and core hosted app behaviors still work.
- Documentation accurately reflects the implemented system.
